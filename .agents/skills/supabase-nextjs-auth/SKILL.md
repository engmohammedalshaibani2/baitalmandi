# Supabase Next.js Session Handling

Related skill: `agent-customization` — use its template and principles when editing this file.

## Summary

This skill captures a repeatable workflow for implementing and debugging Supabase session cookie handling in Next.js (App Router) projects. It describes the step-by-step process to locate, inspect, and fix code that reads session cookies (sb-{projectRef}-auth-token), refreshes tokens, and sets Set-Cookie headers in server middleware and API routes.

## When to use

- You're implementing Supabase authentication with cookies in a Next.js App Router app.
- You're debugging session renewal, token refresh, or cookie propagation between middleware, server components, and API routes.

## Inputs

- Workspace with Next.js App Router code (middleware.ts, server client helpers, API auth routes, login actions).
- Access to supabase client helpers and cookie utilities.

## Output

- A checklist-driven code change set (edits to middleware, server client, and routes) that ensures:
  - Session cookies are read reliably in middleware and server components.
  - Token refresh flows write Set-Cookie headers correctly for the client.
  - Old/duplicate cookies are cleared when creating new sessions.

## Step-by-step Workflow

1. Inspect existing auth-related files: middleware.ts, utils/supabase/server.ts, api/auth/login/route.ts, admin/login/actions.ts.
2. Identify where cookies are read (e.g., `cookies()` from `next/headers` or `Request` headers) and where cookies are written (manual `Set-Cookie` strings or helpers).
3. Confirm cookie names and formats: `sb-{projectRef}-auth-token` (and any `supabase` cookies). Ensure consistent projectRef usage.
4. Verify token refresh logic: determine whether refresh is performed client-side, in middleware, or in API routes.
5. If middleware refreshes tokens, ensure it can send a `Set-Cookie` back: use Response rewriting or Next.js middleware-compatible mechanisms.
6. If API routes set cookies, ensure they use correct `Set-Cookie` attributes (Path, HttpOnly, Secure, SameSite, Expires/Max-Age) and do not conflict with middleware behavior.
7. Add tests / manual checks: log cookie values at each stage, run auth flow, verify cookies on browser and server calls.
8. Iterate on edge cases: concurrent sessions, expired refresh tokens, and cross-subdomain cookie needs.

## Decision Points & Branching

- Where to refresh tokens:
  - Middleware: Centralized, runs on every request; needs careful handling to return Response with Set-Cookie.
  - API route: Fine for explicit flows (login/refresh endpoints); simpler to set cookies on Response.
  - Server component: Useful to read cookies using `cookies()` but cannot directly set Set-Cookie for subsequent client requests.
- Cookie naming:
  - Use Supabase defaults when possible; avoid duplicating cookie names across helpers.

## Quality Criteria / Completion Checks

- After changes, the following must hold:
  - Browser receives updated `Set-Cookie` when a refresh occurs.
  - Server-side code reading cookies sees the refreshed token immediately on subsequent requests.
  - No duplicate or stale `supabase` cookies remain after login or logout flows.
  - Cookies have secure attributes appropriate for your environment (`Secure` in prod, `HttpOnly` for tokens).
